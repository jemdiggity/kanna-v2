# i18n Design Spec

## Overview

Add internationalization to Kanna's Vue frontend, supporting English, Japanese, and Korean. All user-facing strings are currently hardcoded in English across 23 Vue components and several composables. Rust backend has no user-facing strings — i18n is entirely a frontend concern.

## Library

**vue-i18n** in Composition API mode (`legacy: false`). Chosen for its Vue 3 ecosystem support, `useI18n()` composable, and reactive locale switching. No build-time compilation plugin — overkill for ~200 static strings.

## File Structure

```
apps/desktop/src/
  i18n/
    index.ts              # createI18n setup, export plugin
    locales/
      en.json             # English (source of truth)
      ja.json             # Japanese
      ko.json             # Korean
```

## Plugin Setup

`apps/desktop/src/i18n/index.ts`:

```typescript
import { createI18n } from 'vue-i18n'
import en from './locales/en.json'
import ja from './locales/ja.json'
import ko from './locales/ko.json'

const i18n = createI18n({
  legacy: false,
  locale: 'en',
  fallbackLocale: 'en',
  messages: { en, ja, ko }
})

export default i18n
```

Registered in `main.ts` via `app.use(i18n)` before `app.mount()`.

All three locale files eagerly imported — they're small enough that lazy loading adds unnecessary complexity.

## Locale Persistence

Stored in the existing SQLite `settings` table as `key: 'locale'`, `value: 'en' | 'ja' | 'ko'`.

- Default setting `{ key: 'locale', value: 'en' }` added to `runMigrations` in `stores/db.ts` (alongside the existing `INSERT OR IGNORE` defaults at line 86-88).
- On startup (after migrations), read with `getSetting(db, 'locale')` and set `i18n.global.locale.value`. This happens in `App.vue`'s `onMounted` after DB initialization.
- On user change via PreferencesPanel, the existing `emit('update', key, value)` pattern handles persistence — the parent (`App.vue`) calls `setSetting(db, 'locale', newValue)`. The handler also sets `i18n.global.locale.value` to update the UI reactively.
- No app restart required — vue-i18n reactively updates all `$t()` / `t()` calls.

## Preferences UI

Add a language dropdown to `PreferencesPanel.vue` using the existing emit pattern:

```html
<div class="pref-row">
  <label>{{ $t('preferences.language') }}</label>
  <select
    :value="preferences.locale"
    @change="emit('update', 'locale', ($event.target as HTMLSelectElement).value)"
  >
    <option value="en">English</option>
    <option value="ja">日本語</option>
    <option value="ko">한국어</option>
  </select>
</div>
```

The `locale` field is added to the `preferences` prop interface. The parent component handles the `update` event by calling `setSetting` (DB persistence) and setting `i18n.global.locale.value` (reactive UI update) — same pattern as the existing `suspendAfterMinutes`/`killAfterMinutes`/`ideCommand` fields.

Language names displayed in their own language (standard practice — users can find their language regardless of current locale).

## Keyboard Shortcuts & Translation Keys

The `shortcuts` array in `useKeyboardShortcuts.ts` is a **module-level constant** — it is initialized at import time, outside any Vue component setup context. `useI18n()` cannot be called at module scope (it requires a component setup context). The array is also consumed by non-reactive functions (`isAppShortcut`, `getShortcutGroups`).

**Solution:** Store translation keys (not translated strings) in the shortcuts array. Resolve them at render time.

```typescript
// ShortcutDef changes:
interface ShortcutDef {
  action: ActionName
  /** Translation key for the display label */
  labelKey: string
  /** Translation key for the group name */
  groupKey: string
  // ... rest unchanged
}

// The module-level array stores keys:
export const shortcuts: ShortcutDef[] = [
  { action: "newTask", labelKey: "shortcuts.newTask", groupKey: "shortcuts.groupTasks", ... },
  // ...
]
```

`getShortcutGroups()` is refactored to accept a `t` function parameter:

```typescript
export function getShortcutGroups(t: (key: string) => string) {
  const groupOrder = ["shortcuts.groupTasks", "shortcuts.groupNavigation", "shortcuts.groupViews", "shortcuts.groupHelp"]
  const map = new Map<string, { keys: string; action: string }[]>()
  for (const def of shortcuts) {
    if (!map.has(def.groupKey)) map.set(def.groupKey, [])
    map.get(def.groupKey)!.push({ keys: def.display, action: t(def.labelKey) })
  }
  return groupOrder.filter((g) => map.has(g)).map((g) => ({ title: t(g), shortcuts: map.get(g)! }))
}
```

Consumers call it with `t` from their component context:

```typescript
// In KeyboardShortcutsModal.vue
const { t } = useI18n()
const groups = computed(() => getShortcutGroups(t))
```

The `groups` assignment becomes a `computed()` so it reactively updates on locale change. `isAppShortcut()` is unaffected — it only checks key bindings, not labels.

Similarly, `useShortcutContext.ts` context titles become translation key lookups:

```typescript
// Before: const titles: Record<ShortcutContext, string> = { main: "Main Shortcuts", ... }
// After: resolved via t('shortcutContexts.main') at call site
```

## Toast Messages

`useToast` is a module-level composable that accepts plain strings. Toast messages are emitted from Vue components (via `App.vue`) and Pinia stores (via `kanna.ts`). Pinia store actions cannot call `useI18n()` inside their action bodies.

**Solution:** Callers pass translated strings to `useToast`. Two patterns:

1. **In Vue components** (App.vue, etc.): Call `t()` before passing to toast:
   ```typescript
   const { t } = useI18n()
   toast.error(t('toasts.taskCreationFailed'))
   ```

2. **In Pinia stores**: Access the i18n instance directly via the exported module:
   ```typescript
   import i18n from '../i18n'
   const { t } = i18n.global
   toast.error(t('toasts.taskCreationFailed'))
   ```

The `useToast` API itself remains unchanged — it continues to accept plain strings. The translation happens at the call site.

## CJK Font Rendering

The app's font stack (`-apple-system, BlinkMacSystemFont, "SF Pro Text", "Helvetica Neue", sans-serif`) has no explicit CJK fonts. On macOS, the OS font fallback chain handles CJK rendering via Hiragino Kaku Gothic ProN (Japanese) and Apple SD Gothic Neo (Korean) — this produces good results on macOS.

Since Kanna is macOS-only, this is acceptable. No font stack changes needed. If the app ever targets other platforms, explicit CJK font families should be added.

## Component Migration

### Templates

```html
<!-- Before -->
<h2>New Task</h2>
<button>Cancel</button>

<!-- After -->
<h2>{{ $t('tasks.newTask') }}</h2>
<button>{{ $t('actions.cancel') }}</button>
```

### Migration Order

1. `i18n/index.ts` + locale files + `main.ts` registration — infrastructure
2. `stores/db.ts` — add locale default setting
3. `useKeyboardShortcuts.ts` — change `label`/`group` to `labelKey`/`groupKey`, refactor `getShortcutGroups` to accept `t`
4. `useShortcutContext.ts` — replace hardcoded context titles with translation keys
5. `App.vue` — load locale on startup, handle locale update from PreferencesPanel, migrate command palette strings
6. `PreferencesPanel.vue` — add language dropdown, migrate existing strings
7. `KeyboardShortcutsModal.vue` — use `computed(() => getShortcutGroups(t))`, migrate template strings
8. `Sidebar.vue`, `ActionBar.vue` — always visible, migrate template strings
9. All remaining modals and components — migrate template strings
10. Pinia stores (`kanna.ts`) — use `i18n.global.t` for toast messages

## Scope

- Static strings only — no interpolation, pluralization, or date/number formatting.
- Frontend only — no Rust changes needed.
- No lazy loading — all locales bundled.
- No RTL support needed (en, ja, ko are all LTR).
- CJK rendering relies on macOS OS-level font fallback (acceptable for macOS-only app).
